# Бизнес-правила

## Производитель по реестру ГИСП

- Поиск выполняется только для позиции без товарного знака и без уже указанного производителя.
- Ключ поиска — точный реестровый номер ПП РФ №719 после удаления пробелов и разделителей.
- Источник — официальная XLSX-выгрузка действующих записей Реестра российской промышленной продукции ГИСП.
- При точном совпадении поле производителя заполняется значением `Предприятие` из реестра.
- Если запись не найдена или ГИСП временно недоступен, закупка не падает: позиция отправляется на ручную проверку.

## Идентификация закупки

Закупка идентифицируется только по номеру закупки. Номер всегда хранится и обрабатывается как строка, чтобы не терять ведущие нули и не получать экспоненциальный формат.

## Заполняемые поля

Приложение автоматически заполняет поля `Победная ставка`, `Кто выиграл`, `ИНН победителя`, `Поставляемый товар`. Дополнительно ведутся служебные поля: источники документов, автостатус, даты проверок, текущая цена контракта, признак внимания, ручная фиксация и первоначальный победитель.

## Победитель и ставка

Главный источник победителя и победной ставки — протокол подведения итогов. Денежные значения хранятся в базе как `Decimal`, а в Google Sheets записываются числом без символа рубля.

Если процедура завершилась заключением контракта с единственным поставщиком, победитель, ИНН и ставка берутся из заключённого контракта. Источником считается контракт.

Если первоначальный победитель уклонился и контракт заключён с другим участником, в рабочие поля записывается фактический поставщик, первоначальный победитель сохраняется в служебной колонке, включается `Ко вниманию`, а закупка попадает на лист `Требуется проверка`.

ИНН считается подтверждённым только при корректной контрольной сумме: 10 цифр для организации, 12 цифр для ИП или физического лица. Победитель без корректного ИНН отправляется на ручную проверку.

## Нестандартные результаты

Поддерживаются буквальные статусы:

- `Закупка отменена`
- `Закупка не состоялась`
- `Не подано ни одной заявки`
- `Все заявки отклонены`
- `Подана одна заявка`
- `Контракт не заключён`
- `Победитель уклонился`
- `Результаты аннулированы`

Для таких случаев без последующего контракта заполняются: `Победная ставка` = `-`, `Кто выиграл` = статус, `ИНН победителя` = `-`, `Поставляемый товар` = `-`.

## Поставляемый товар

Товар извлекается только после появления контракта и спецификации к нему. Нельзя считать исходное техническое задание или описание объекта закупки фактически поставляемым товаром.

Источники товара по приоритету: спецификация контракта, структурированная часть контракта, приложения к контракту, официальные изменения спецификации. Поддерживаются HTML, XML, PDF, DOCX и XLSX. OCR не включён в MVP и должен использоваться только как резервный механизм для сканов.

Для позиции извлекаются страна, производитель, торговая марка, модель, реестровый номер Минпромторга по ПП РФ № 878, количество и единица измерения. Реестровый номер запрещено подбирать по названию, модели или производителю; он сохраняется только если найден в документах конкретной закупки.

## Ручные изменения

После автоматической записи приложение сохраняет снимок поля и нормализованный хеш. При следующем запуске поле обновляется только если текущее значение совпадает с последним автозначением. Если сотрудник изменил значение вручную, приложение не перезаписывает его и добавляет конфликт в лист проверки.

## Лист проверки

Лист `Требуется проверка` создаётся автоматически. В него попадают закупки с недостаточными данными, конфликтами ручных значений, расхождением победителя и поставщика, некорректным ИНН, отсутствующей спецификацией при найденном контракте и ошибками парсинга.

Когда в листе проверки статус меняется на `Проверка завершена`, синхронизация закрывает соответствующую запись очереди в PostgreSQL.

## Повторная обработка

Повторные запуски идемпотентны: строка каждый раз ищется заново по номеру закупки, снимки полей защищают ручные значения, а аудит хранит историю изменений бессрочно.
